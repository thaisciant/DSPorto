# Research: Button Component — DS Essential 2.0

**Date**: 2026-06-15
**Feature**: specs/001-button-component/spec.md

---

## Decision 1: Cross-platform architecture pattern

**Decision**: MVVM for Web (React) and iOS (SwiftUI); MVI for Android (Jetpack Compose).

**Rationale**: React's unidirectional data flow and SwiftUI's `@ObservableObject`
map naturally to MVVM. Jetpack Compose's `StateFlow` + sealed class ecosystem
makes MVI (Model–View–Intent) the idiomatic choice for Android. Both patterns
satisfy constitution Principle III and keep Views stateless.

**Alternatives considered**:
- **Redux (web)**: Over-engineered for a single stateless component; adds boilerplate
  without benefit at this scope.
- **UIKit (iOS)**: Excluded — the spec targets iOS 16+ and SwiftUI is the strategic
  platform direction for DS 2.0.
- **MVP (Android)**: Valid but MVI provides better unidirectional guarantees and
  is the recommended pattern for Compose-first projects in 2026.

## Decision 2: Token delivery mechanism per platform

**Decision**:
- Web: CSS Custom Properties consumed via `var(--btn-*)` in component styles,
  generated by Style Dictionary from `channel.component.button.*` JSON.
- iOS: `ButtonTokens.swift` — static `let` constants in a `ButtonTokens` enum,
  generated by Style Dictionary.
- Android: `ButtonTokens.kt` — `object ButtonTokens` with `val` constants,
  generated by Style Dictionary. XML resources also generated for View system
  consumers.

**Rationale**: Each platform uses its idiomatic token delivery format. The
`@ps/design-tokens` package is the build-time source; no runtime token fetching
is needed. This satisfies constitution Principle I (no hard-coded values) and
Principle V (token naming convention).

**Alternatives considered**:
- **Inline style objects (React)**: Rejected — prevents CSS cascade, dark mode via
  `[data-theme]` attribute, and media query responsiveness.
- **Hardcoded values with comments**: Rejected — violates constitution Token
  Integrity gate and breaks the Figma → code sync pipeline.

## Decision 3: Testing strategy per platform

**Decision**:
- **Web**: Vitest + React Testing Library for unit/integration; axe-core for
  automated WCAG audit; Storybook play functions for visual interaction testing;
  snapshot via Storybook Chromatic or jest-image-snapshot.
- **iOS**: XCTest for unit tests on ViewModel; swift-snapshot-testing for visual
  regression across all 72 state permutations.
- **Android**: JUnit 5 for ViewModel/Intent unit tests; Paparazzi for snapshot
  testing of Composables without a device/emulator.

**Rationale**: Each tool is the community-standard for its platform. TDD is
enforced by writing test stubs before implementation files (constitution Principle
IV). 80% floor verified in CI using Istanbul (web), Xcode coverage (iOS), and
JaCoCo (Android).

**Alternatives considered**:
- **Detox / Maestro (E2E)**: Excluded from this feature scope — component-level
  testing is sufficient; E2E tests live in the product app layer.
- **Enzyme (web)**: Deprecated; RTL is the current standard.

## Decision 4: Responsive breakpoint implementation (web)

**Decision**: CSS media query using `channel.breakpoint.tablet` (768 px). Below
this breakpoint, the Button stretches to 100% of its container width via a CSS
class modifier (`.btn--full-width`) applied by the parent layout or a `fullWidth`
prop. The Button does not self-detect viewport width at runtime.

**Rationale**: A component that reads its own viewport size violates single
responsibility (constitution Principle I — SRP). The consuming layout controls
full-width behaviour via a prop or class; the Button only needs to know how to
render itself when `fullWidth` is set.

**Alternatives considered**:
- **ResizeObserver inside Button**: Rejected — couples the component to layout
  concerns; makes testing harder; breaks SSR.
- **CSS container queries**: Viable but introduces a wider browser-support concern
  than media queries for this token. Deferred to v2.

## Decision 5: Loading state indicator

**Decision**: A spinner (CSS animation or native ActivityIndicator) replaces the
leading-icon slot while the label remains visible, and the button is set to
`disabled` / `pointerEvents: none`. The spinner uses the foreground token colour
of the active variant.

**Rationale**: Keeping the label visible during loading prevents layout shift
(CLS). The icon slot is the least disruptive location for a spinner. Disabling
pointer events prevents double-submission.

**Alternatives considered**:
- **Replace label with spinner**: Rejected — causes CLS; label text anchors
  the user's mental model of the pending action.
- **Overlay with opacity 0.5**: Rejected — opacity change alone does not
  communicate "in progress" clearly enough; also fails contrast gate in some themes.

## Decision 6: Focus ring implementation

**Decision**: CSS `outline` (not `box-shadow`) with `outline-offset: 2px` on web.
On iOS, `accessibilityFocus` styling with a 2 pt border. On Android, the default
Compose focus indicator is overridden with a custom `FocusRequester` + `Indication`
drawing a 2 dp ring in `ButtonTokens.Color.Border.FocusRing`.

**Rationale**: `outline` is not clipped by `overflow: hidden` unlike `box-shadow`.
Using `outline` satisfies WCAG 2.4.7 without requiring JS. The ring uses the
`theme.color.interactive.focus` token (`#5B9BFF`) which has a 3:1 contrast
ratio against both light and dark backgrounds (WCAG 2.1 AA for UI components).

**Alternatives considered**:
- **box-shadow**: Rejected — clipped inside rounded containers; not an actual
  focus indicator for AT.
- **Custom JS focus trap**: Rejected — native browser/OS focus management is
  always preferred.
